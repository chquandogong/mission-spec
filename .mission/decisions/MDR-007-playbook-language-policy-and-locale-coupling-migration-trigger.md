# MDR-007: Playbook language policy and locale-coupling migration trigger

**Date:** 2026-04-17
**Status:** Active
**Version:** 1.16.5
**Author:** Dr. QUAN

Third real-world use of `ms-decide` on its own project (after MDR-005, MDR-006). Section bodies below were authored by the maintainer.

## Context

`scripts/verify-registry.js` (v1.16.0, D-1 closure)의 `PLAYBOOK_PATTERNS` 배열은 **한국어 정규식 6개**로 REBUILD_PLAYBOOK.md의 수치 주장을 매칭한다:

```js
/(\d+)개 모듈/
/public API (\d+)개 함수/
/skill (\d+)개/
/(\d+)개 플랫폼/
/\((\d+) test files?\)/
/현재 기준 (\d+) tests? 전수 통과/
```

이 중 4개는 한국어 조사("개 모듈", "개 플랫폼")에 결합되어 있고, 1개는 한국어 표현("현재 기준 ... 전수 통과")에 결합되어 있으며, 1개(`(N test files)`)만이 locale-independent다. 현재 REBUILD_PLAYBOOK.md가 한국어이므로 문제는 없으나, **playbook이 영문/중문으로 번역되면 정규식이 조용히 no-op이 되어 drift를 감지하지 못한다**.

세 독립 리뷰에서 이 지점이 MDR-레벨 결정으로 승격될 필요가 지적됨:

- Claude v1.16.0 SNAPSHOT §5.3 (고유 발견): "i18n 전략과 drift detector 설계가 충돌한다. README가 3개 언어 유지되고 ms-decide skill이 3개 언어로 배포되는 프로젝트가, 가장 중요한 prose-asset인 REBUILD_PLAYBOOK을 한국어 패턴에 hard-coupling했다."
- Codex v1.16.0 §6 (독립 확증): "playbook이 영문/중문으로 바뀌면 drift detector가 조용히 no-op이 될 수 있다."

더 큰 맥락:

- v1.8.0부터 `README.md`, `skills/ms-*/SKILL.md`는 **ko/zh/en 삼국어 유지** — 설치 사용자를 삼국어 커뮤니티로 가정한다.
- 그러나 `.mission/` 하위 descriptive asset은 **한국어 전용**이다 — 역사적 관성으로 유지됐고 지금까지는 문제된 적 없었다.
- 본 MDR은 이 비대칭의 **명시적 인정 또는 교정**을 결정한다.

## Decision

**Hold + Trigger 모델**: 현재 상태를 **명시적 정책으로 추인**하고, 미래 변경 조건을 트리거로 기록한다.

### 현재(v1.16.x ~ v1.x) 정책

1. `.mission/reconstruction/REBUILD_PLAYBOOK.md`는 **한국어 canonical**을 유지한다. 번역본은 생성하지 않는다.
2. `PLAYBOOK_PATTERNS`는 한국어 결합을 유지한다 — 현재 playbook body와 매치하므로 drift 감지가 실제로 작동한다.
3. v1.8.0 trilingual 원칙의 스코프는 **consumer-facing artifacts**(README, skills, marketplace 설명)에 한정한다. `.mission/`는 **maintainer-facing artifact**이므로 이 원칙 밖이다.
4. 이 경계는 **관성이 아니라 결정**이다 — 본 MDR로 그 지위를 얻는다.

### 마이그레이션 트리거

다음 중 **하나**라도 발생하면 즉시 MDR-008(또는 본 MDR amendment)로 구체적 마이그레이션 경로를 확정한다:

- **T-1**: 비-한국어 maintainer가 joining하여 `.mission/` 편집 권한을 받는 경우 (bus factor 변화 — 한국어 읽기/쓰기 가정이 깨지는 시점).
- **T-2**: 외부 adopter(E-4 closure 맥락)가 영어 또는 중국어 playbook을 요청하는 경우.
- **T-3**: 어떤 이유로든 REBUILD_PLAYBOOK.md 번역이 제안되는 경우 — 누가 제안하든, 왜 제안하든.
- **T-4**: `.mission/` 하위의 다른 descriptive asset(CURRENT_STATE, VERIFICATION_LOG 등)이 번역되는 경우 — playbook만 한국어 유지하는 것이 기형이 되는 시점.

트리거 발동 시 기본 마이그레이션 경로는 §Alternatives #1(YAML frontmatter for numeric claims). 발동 시점에 재검증 후 채택.

### 예방 조치(현재 구현)

- `scripts/verify-registry.js`의 `PLAYBOOK_PATTERNS` 상수 바로 위에 본 MDR 참조 주석을 추가해, 다음 사람이 번역을 고민할 때 이 결정의 존재를 찾을 수 있게 한다. (v1.16.5 이 릴리스에서 함께 적용)
- REBUILD_PLAYBOOK.md 상단에 한 줄 메타 주석 추가(옵션; 이번 릴리스에서는 skip — 본문 편집이 registry:check의 PLAYBOOK_PATTERNS에 영향 줄 수 있어 별도 확인 필요).

## Rationale

1. **현재 위험은 0** — playbook이 한국어고, 패턴이 한국어며, drift 감지가 실제 작동 중. 실측 0건의 silent-pass 상황.

2. **선제 마이그레이션의 비용 > 선제 방어의 효용** — 오늘 하루(2026-04-17) 안에 3개 MDR(005, 006, 007) 작성이 이미 governance 피로의 경계. 추가로 frontmatter 스키마 + migration + deprecation까지 구현하는 것은 **현실 소비자 없는 상태의 over-engineering**.

3. **Trilingual은 "실 사용자 있는 표면"의 원칙이었음** — v1.8.0의 trilingual은 설치 사용자(skills)와 README 독자를 대상으로 했다. 이들은 실존 다국어 사용자 집단. 반면 REBUILD_PLAYBOOK은 "코드 잃고 재구성하는 maintainer"를 위한 것 — 현재 maintainer는 저자 1인(한국어 사용자).

4. **YAML frontmatter 경로는 트리거 시점의 "기본값"으로 이미 준비** — `.mission/traceability/TRACE_MATRIX.yaml`의 `test_coverage.cases` 필드가 선례. verify-registry.js는 이미 YAML 파싱을 하고 있으므로, playbook에 frontmatter를 추가해 거기서도 numeric claim을 읽는 패턴 확장은 기계적으로 작은 변경.

5. **트리거 서술이 정책 자체의 stale을 방지** — "언제 다시 볼 것인가"가 기록되지 않은 MDR은 오래된 채 방치되기 쉽다. T-1~T-4 중 어느 하나라도 발생하면 본 MDR이 **자동으로 revisit 대상**이 된다는 것을 명시함으로써 governance drift 자체를 막는다.

## Consequences

1. `scripts/verify-registry.js`: `PLAYBOOK_PATTERNS` 상수 위에 참조 주석 추가(**이 릴리스에서 함께 적용** — docs-only commit이 아닌 "docs + 1-line comment")만 실제 코드 수정. 기능 변경 없음.

2. `docs/`, README, CONTRIBUTING: `.mission/` 하위 asset이 한국어임을 **의도된 정책**이라고 인정하는 한 줄 추가 여부는 별도 검토. 본 MDR 자체가 공식 답이므로 추가 문서화 없이 `MDR-007` 참조로 충분.

3. 본 MDR은 MDR-005(scope), MDR-006(grade), MDR-007(locale)이라는 세 축으로 governance 레이어를 완성 — v1.x contract가 이제 **기술 + scope + grade + locale** 네 축 모두 명문화된 MDR 커버리지를 가진다.

4. 트리거가 발동하면 본 MDR은 Status를 **Superseded by MDR-008**로 바꾸거나, 같은 번호 아래 Rev.2로 amendment. Mission-history에 둘 중 어느 방식을 취했는지 기록한다.

## Alternatives Considered

1. **YAML frontmatter for numeric claims (Option 3, 이번에 채택 안 함)** — `.mission/reconstruction/REBUILD_PLAYBOOK.md` 상단에 `---` frontmatter 블록을 추가하고 `moduleCount: 18`, `testCount: 237` 등을 구조화 필드로 기록. verify-registry가 이 필드를 우선 읽고, 없으면 legacy prose pattern으로 fallback. **가장 깨끗한 설계**지만 현재 소비자가 없어 비용 대비 효용 낮다. 트리거 발동 시 채택 기본값.

2. **Per-locale pattern sets** — `PLAYBOOK_PATTERNS_KO`, `PLAYBOOK_PATTERNS_EN`, `PLAYBOOK_PATTERNS_ZH`를 각각 유지하고 body language 감지 후 매칭. 언어별 패턴 추가 시 linearly scaling. 하지만 playbook이 일본어가 되면 또 추가해야 하고, mixed-language playbook은 여전히 위험. 거부.

3. **Translate REBUILD_PLAYBOOK.md to English now** — 가장 짧은 해결책이지만 **현재 사용자 없음**. 저자 단독 maintainer가 한국어로 쓰고 한국어로 읽는 파이프라인을 "있지도 않은 미래 영어 사용자"를 위해 영어화하는 것은 premature optimization. 거부.

4. **Drop PLAYBOOK_PATTERNS entirely and rely on TRACE_MATRIX YAML only** — verifier의 prose coverage 포기. 현재 verifier는 REBUILD_PLAYBOOK prose + TRACE_MATRIX YAML + CURRENT_STATE 세 축 커버. 한 축 포기는 detection 범위 축소. 거부.

5. **본 MDR을 skip하고 `.mission/` 전체를 trilingual로** — trilingual 원칙의 확장. 하지만 `.mission/` 자산은 "재구성"에 특화된 매우 긴 prose/YAML. 삼국어 유지 비용이 실재 사용자 없음과 비례 안 한다. 거부.

6. **MDR 작성을 다음 사이클로 연기** — "실제 문제 발생 전까지 기다린다". 현재 매일의 리뷰 사이클에서 "이 결정을 기록 안 하면 다음 리뷰가 또 지적한다"는 것이 관찰된다(Claude SNAPSHOT §5.3 + Codex §6). **거부** — MDR-005/006 작성 리듬을 유지하는 것이 governance 효율.
