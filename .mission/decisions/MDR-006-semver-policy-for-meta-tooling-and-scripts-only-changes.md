## Context

v1.9 이후(2026-04-17 당일) 오늘 안에 총 15차례 semantic release가 있었으나, 각 릴리스의 SemVer 등급(major/minor/patch) 판정은 **명시 규칙 없이 직관에 의존**했다. 실제 판정 기록:

| 버전    | 등급      | 변경 성격                                                                      |
| ------- | --------- | ------------------------------------------------------------------------------ |
| v1.9.0  | minor     | LICENSE, CHANGELOG, CONTRIBUTING, 2 workflow, coverage 80%                     |
| v1.10.0 | minor     | 새 script `architecture-extractor` + `arch:sync`/`check`/`verify` 3 npm script |
| v1.11.0 | minor     | 새 script `plugin-validator` + `plugin:verify` npm script                      |
| v1.12.0 | minor     | 새 CLI `bin/mission-spec.js` + 영문 휴리스틱 25 동사                           |
| v1.13.0 | minor     | 새 `release.yml` workflow, mission-history 영문 정규화                         |
| v1.13.1 | patch     | 대칭 drift detection(기존 arch:verify 확장) + publish ref 가드                 |
| v1.14.0 | minor     | `ms-decide` + migration + reconstruction-verifier + 3 신규 플랫폼              |
| v1.14.1 | patch     | Unicode-aware slugify bugfix                                                   |
| v1.14.2 | patch     | MDR-005 작성 + README/marketplace 동기화                                       |
| v1.14.3 | patch     | arch:verify에 package_exports key-set diff 추가(기존 script 확장)              |
| v1.15.0 | **minor** | 새 script `bump-metadata.js` + `metadata:sync`/`metadata:check` npm scripts    |
| v1.16.0 | **minor** | 새 script `verify-registry.js` + `registry:check`/`registry:list` npm scripts  |
| v1.16.1 | patch     | `release.yml`에 cold-build gate step 추가 (CI-only)                            |
| v1.16.2 | patch     | arch:verify value-shape 확장 + registry:check CURRENT_STATE 확장 (기존 script) |
| v1.16.3 | patch     | bump-metadata.js에 Title 라인 sync 추가 (기존 script 확장)                     |

핵심 불일치: **v1.15.0 / v1.16.0은 minor**였는데 **v1.16.2 / v1.16.3은 patch**였다. 전자는 새 top-level npm script를 추가했고 후자는 기존 script의 기능을 확장했다 — 이 구분은 실제로 둘을 다르게 대하는 근거가 되지만, 본 MDR 이전에는 **그 근거가 어디에도 서면화되어 있지 않았다**.

세 독립 리뷰 중 Claude v1.16.0 SNAPSHOT §5.1, 같은 문서 §6 E-3이 이 정책 공백을 명시 지적. Codex v1.16.0 §7에서도 "Low-Medium" 우선순위로 동일 지적:

> "scripts-only 릴리스의 SemVer 기준이 아직 모호하다 ... 외부 채택자가 생기면 버전 해석 비용이 커진다."

v1.16.1 / v1.16.2 / v1.16.3 세 릴리스 모두 "CI-only change" / "scripts-only extension"의 근거로 patch를 고른 선례가 쌓였으므로, 본 MDR이 **후행 서술**이 아니라 **선행 기록 + 선례 인정** 성격을 띤다.

## Decision

Mission Spec의 SemVer 등급 판정은 **변경 대상의 소비자 가시성(consumer surface)** 을 기준으로 한다. 판정 기준 3등급:

### §MAJOR — breaking consumer surface

- `mission.yaml` 또는 `mission-history.yaml` 스키마의 breaking change (기존 필수 필드 제거, 타입 변경, 의미 변경)
- `src/index.ts` public API 함수 제거 또는 시그너처 breaking change
- 기존 skill 또는 플랫폼 어댑터 제거
- 기존 CLI subcommand 제거 또는 호환 breaking change
- `.mission/` 구조 변경으로 하위 소비자(verify-registry / ms-context)가 읽을 수 없게 되는 경우

### §MINOR — additive consumer surface

- 새 public API 함수 또는 타입 export (`src/index.ts` 추가)
- 새 skill 패키지 (`skills/ms-*/`)
- 새 플랫폼 어댑터 (`src/adapters/platforms.ts`의 `convertTo*` 추가)
- 새 CLI subcommand
- 새 `package.json.exports` subpath
- **새 top-level `package.json.scripts` 엔트리** — 외부에서 `npm run <new-name>` 호출 가능한 새 진입점
- 새 workflow **job** 추가 (릴리스 파이프라인의 새 의존 단계)
- 기존 validator의 필수-exit-code 행동 변경 — "이전에 경고였던 것이 이제 실패"

### §PATCH — no consumer surface change

- 기존 script의 기능 확장 — **`package.json.scripts` 이름이 그대로**이고 추가 플래그/환경변수로 새 행위를 노출하지 않는 경우 (v1.14.3, v1.16.2, v1.16.3이 해당)
- 기존 validator의 drift detection 범위 확장
- CI workflow 편집 — 기존 job에 step 추가 / step 순서 조정 (v1.16.1이 해당)
- bugfix (기능 unchanged)
- docs / MDR / README / CHANGELOG 업데이트
- test-only 추가 또는 리팩터
- `.mission/` descriptive asset 내용 갱신

## Rationale

1. **semver.org의 정의와 일치** — v2.0 시점에 기능을 제거/변경하려면 major이어야 한다는 것은 "내 사용자가 내 변경 때문에 자기 코드를 고쳐야 하는가?"로 결정된다. scripts-only CI 확장은 소비자 입장에서 변화가 없으므로 patch.

2. **실제 관찰된 패턴을 인코딩** — v1.14.3(patch, arch:verify 확장)과 v1.16.3(patch, bump-metadata 확장)은 구조적으로 같다. 같은 것을 다르게 부르는 것은 향후 외부 채택자(E-4)에게 **버전 해석 비용을 전가**한다.

3. **v1.15.0 / v1.16.0 minor 판정의 사후 승인** — 새 top-level npm script(`metadata:sync`/`metadata:check`, `registry:check`/`registry:list`)가 추가됐으므로 minor가 맞다. v1.16.3의 **같은 script가 기능 확장만 받은 것**은 patch — 이 구분이 본 정책으로 명시화된다.

4. **registry:check와의 self-consistency** — v1.14.3에서 arch:verify가 `package_exports`를 공식 검증 대상에 편입했다는 것은 이 exports map이 "소비자 surface"의 일부임을 인정한다는 뜻. 따라서 새 subpath 추가는 minor, 기존 subpath의 value 변경(E-5)은 patch가 자연스럽다.

5. **"15분 작업의 MDR" 원칙 확인** — MDR-005 Rationale에서 자기 교정 루프가 방법론으로 인정됐듯, 정책 MDR도 실제 사용 선례가 생긴 뒤 쓰는 것이 옳다. 본 MDR 작성 시점(v1.16.4 직전)에 v1.16.1-3 세 릴리스가 이미 §PATCH 규칙대로 운영된 상태 — 서술형 기록이 아니라 **3번의 실행 후 정리**다.

## Consequences

1. 차기 릴리스부터 모든 `mission-history.yaml` 엔트리는 **`risk` 필드 끝에 SemVer 등급 근거**를 명시한다. 형식:

   ```
   Semantic version: <major|minor|patch> per MDR-006 §<범주> (<근거 한 줄>).
   ```

   v1.16.1~v1.16.3이 이미 이 패턴의 전구체를 담고 있으므로 retroactive 수정 없음 — v1.16.4부터 정식 참조.

2. v1.16.1 → §PATCH 3 (CI workflow 편집), v1.16.2 → §PATCH 1 (기존 script 기능 확장), v1.16.3 → §PATCH 1. 본 MDR 채택 시점에 **사후적으로 분류가 확정**된다.

3. MDR-005 Consequences와 상보: MDR-005는 "무엇을 허용하는가"(scope), MDR-006은 "허용된 것을 어떤 등급으로 발표하는가"(grade). 두 MDR을 연결해 읽어야 meta-tooling 추가의 full lifecycle(scope check → grade judgment → release)이 담긴다.

4. 외부 채택자가 생긴 이후(E-4)에도 정책이 고정되어 있으므로 downstream이 "0.x API 유동기" 가정 없이도 업그레이드 의사결정 가능. 특히 **`^1.x` 또는 `~1.16` 같은 semver range 지정 시 예측 가능성** 확보.

5. 애매한 경계 케이스(예: "pre-commit 훅에 step 추가"는 §PATCH 3 CI workflow 편집과 §MINOR 6 새 workflow job 사이 어디인가?) 는 다음 사이클에서 등장 시 본 MDR을 amend하는 방식으로 해결. 각 케이스를 history entry `decision` 필드에 남긴다.

## Alternatives Considered

1. **Case-by-case judgment 유지** — 현 상태. 간단하지만 v1.14.3(patch)와 v1.15.0(minor)의 판정 차이가 근거 서면화 없이 유지되어 Codex/Claude 리뷰가 지적한 원인. **거부** — 외부 채택자 발생 시 비일관성의 설명 비용이 크다.

2. **모든 `scripts/` 변경을 minor** — 보수적 해석. 하지만 test-only 또는 CI-only 변경까지 minor로 올리면 **릴리스 빈도 대비 minor의 의미가 깎인다**(하루 15 릴리스 중 10개가 minor가 되는 극단). 거부.

3. **모든 `scripts/` 변경을 patch** — 관대한 해석. 하지만 새 top-level npm script 추가(v1.15.0의 `metadata:sync`)는 실제 사용자 표면 변경이며 minor 처리가 합리적 — 컨슈머가 `npm run metadata:sync`를 자기 CI에 추가할 수 있게 되는 순간이다. 거부.

4. **Commit message 컨벤션(feat/fix/chore)으로 대체** — 컨벤션과 릴리스 등급은 다른 축. `feat` 커밋이 여러 개여도 다음 릴리스가 patch일 수 있다(예: 내부 리팩터만 담은 feat). semver와 commit prefix는 **독립 유지**하고 본 MDR로 semver만 규제. 거부.

5. **Zero-version 정책(v0.x 간주) 유지** — 현재 v1.x를 운영 중이며 mission.yaml v1.0.0 이후 breaking change가 없었으므로 이미 1.x contract가 유효. 이를 0.x로 내리는 것은 **과거 22개 snapshot의 의미를 훼손**한다. 거부.

6. **Trunk-based monotonic minor (매 릴리스를 무조건 minor)** — 일부 프로젝트(예: Chrome)의 정책. 하지만 mission-spec은 library + CLI + skill bundle + Living Asset Registry 네 소비자 표면이 있고, 각각이 서로 다른 속도로 변한다. 하나의 monotonic 클럭은 맞지 않는다. 거부.
