## Context

v1.9부터 v1.14.1 사이(2026-04-17 단일 날짜)에 Mission Spec core(스키마·평가·보고) 외에 다수의 meta-tooling 모듈이 추가되었다.

- **Integrity verifiers**: `architecture-extractor` (v1.10), `plugin-validator` (v1.11), `reconstruction-verifier` (v1.14), `validate-history-commits.js` (v1.7), `arch:verify`의 대칭 drift 감지 (v1.13.1).
- **Living asset maintainers**: `ms-decide` + `generateMdrDraft()` (v1.14), `scripts/snapshot-mission.js` (v1.6), `scripts/generate-changelog.js` (v1.9), `migration.ts` BFS chain (v1.14 — registry 비어있음, schema v2용 선행 준비).
- **Distribution surface**: `bin/mission-spec.js` CLI (v1.12), `.github/workflows/{test,pre-commit-parity,release}.yml` (v1.9, v1.13).

이들은 **"task contract 자체"는 아니지만 contract의 무결성·진화·배포를 지키는 도구**다. 그러나 `MDR-001` "Task Contract Only"는 이 확장들이 MDR-001의 경계를 넘었는지에 대한 판단 기준을 담고 있지 않다.

## Decision

**"Task Contract Only"의 경계를 다음과 같이 정제한다:**

Mission Spec의 허용 scope에는 **task contract 자체**(= `mission.yaml` 스키마 + 평가 엔진 + 보고 도구)와 **그 contract의 무결성·진화·배포를 지키는 meta-tooling**이 포함된다.

**허용 범주 3가지:**

1. **Integrity verifiers** — code ↔ registry drift detection, schema/plugin/history 일관성 검사, reconstruction 가능성 검증.
2. **Living asset maintainers** — history entry / MDR / snapshot / migration authoring 보조, 자동 생성·동기화.
3. **Distribution surface** — CLI, release workflow, platform adapters, marketplace metadata, CHANGELOG 생성.

**범위 밖(금지):**

- task execution, agent orchestration, runtime scheduling.
- session-persistent state / external state management (예: Redis, DB 연결).
- 별도 SaaS / Web UI / dashboard / 외부 API 서비스.
- 에이전트가 "어떻게" 실행할지를 결정하는 런타임 로직.
- 다른 orchestration framework와의 런타임 통합 (어댑터로 **export**는 허용, runtime **import**는 금지).

## Rationale

1. **Contract의 가치는 두 축으로 성립** — "무엇이 변하지 않아야 하는가"(고정)와 "무엇이 바뀌었는가"(추적). 두 축을 유지하려면 **drift 방어선이 필수**이며, 방어선 없는 contract는 24시간 안에 stale이 된다. Living Asset Registry 패러다임(v1.5.0, `MSC-2026-04-08-001`)이 이 확장을 예고했다.

2. **MDR-001의 원본 의도는 "orchestration 내장 거부"였지 "운영 인프라 포기"가 아니었음** — MDR-001 "Decision" 본문은 _"Orchestration, scheduling, runtime management는 scope 밖이다"_ 이며, drift detection / migration / reconstruction verification / MDR authoring은 orchestration이 아니다. 본 MDR은 이 해석의 **공식화**이다.

3. **Codex Rev.2 리뷰 통찰(2026-04-17)** — _"문제의 성격이 구현이 비어 있다에서 운영 규율과 증명이 뒤처진다로 이동했다."_ 이 국면에서 meta-tooling은 contract의 후속 생명선.

4. **실증** — v1.10 `architecture-extractor`는 첫 실행에서 plugin.json drift를 잡아냈고, v1.13.1에서 그 검증기 자체를 교정했으며(대칭 drift 감지), v1.14.1에서 `ms-decide`의 Unicode slug 허점을 자체 발견했다. **meta-tooling이 자기 교정 루프로 기능**함이 세 번에 걸쳐 검증됨.

5. **의존성 원칙의 상보성** — 이 모든 확장은 runtime dependency를 늘리지 않았다 (`ajv` + `yaml` 2개 유지). `MDR-003` "Minimal Dependencies"와 공존 가능 — meta-tooling 확장이 core 철학을 훼손하지 않는다는 구조적 증거.

## Consequences

**허용되는 후속 확장 (예시 — `docs/PROJECT_REVIEW_V1.14.1_2026-04-17.md` D섹션):**

- `scripts/verify-registry.js` (D-1) — `.mission/` 서술 자산의 **내용** freshness 검증 (현재는 경로 존재만).
- package.json.exports ↔ API_REGISTRY.yaml.package_exports 동기 검증 (D-2).
- `scripts/bump-metadata.js` (D-3) — patch release 시 `.mission/` 버전 헤더 auto-sync.
- `evals/<name>/*.result.yaml` 다중 verdict 집계 — 3-벤더 adversarial 리뷰 방법론을 자기 도구가 지원하도록.
- Reconstruction verifier `--cold-build`의 release workflow 편입 — bus factor=1 방어선 자동화.

**금지되는 방향:**

- agent execution loop 내장 ("mission.yaml을 받아 실제로 task를 실행").
- mission 상태의 session-persistent 저장소.
- Web UI / dashboard / 외부 API 호출 기반 기능.
- 다른 orchestration framework와의 런타임 import 결합.

**`MDR-001`과의 관계:**

MDR-001의 원문은 그대로 유지한다 — supersede 아님. MDR-005는 **해석 레이어**로 기능하며, 향후 meta-tooling 제안은 본 MDR의 허용 범주 3가지를 기준으로 검토된다. 두 MDR 모두 `Active` 상태로 공존한다.

**거버넌스 효과:**

- v2.0 목표에서 meta-tooling 범위 판정 기준이 명시됨.
- 새로운 검증기 / 어댑터 / authoring skill 제안은 "§Decision 허용 범주에 해당하는가"를 먼저 답해야 함.
- 범위 밖 기능 제안은 별도 프로젝트로 분리 권고 (예: 에이전트 실행기는 `mission-spec-runner` 같은 sibling repo).

## Alternatives Considered

1. **MDR-001 본문을 개정(supersede)** — rejected. 원본 MDR의 integrity 유지 + supersession 체인 추적 용이 + 현 해석이 MDR-001의 의도에 부합하므로 refinement가 정확.

2. **Meta-tooling을 별도 package(`@mission-spec/tools`)로 분리** — rejected. runtime deps는 여전히 `ajv` + `yaml` 2개뿐. bundle 크기 비용 < **조립 복잡성** 비용. 자기 프로젝트 도그푸딩에도 제약 추가.

3. **모든 meta-tooling을 내부 `scripts/*.js`로만 유지하고 public API 제외** — rejected. 3rd-party가 자기 Mission Spec 프로젝트에서 `verifyReconstructionReferences()`, `validatePlugin()`, `extractArchitecture()`, `generateMdrDraft()`를 쓸 수 있어야 task-contract 도구로서 완결된다. registry/validator 없이는 Mission Spec을 쓸 의미가 절반으로 줄어든다.

4. **확장을 v2.0 이후로 연기** — rejected. v1.8 리뷰의 23개 이정표 중 다수가 meta-tooling 영역이었다. 연기하면 **v2.0 자체가 meta-tooling 정체기**가 된다.

5. **허용 범주를 더 넓게 정의(예: "contract 주변 모든 것")** — rejected. 경계가 없으면 MDR-001 존재 가치가 무의미해진다. 3범주 + 명시적 금지 목록이 최소한의 운영 가능한 가드레일.
