## Context

After 7 releases on 2026-04-21 (v1.16.16 → v1.19.0 closing the qmonster B-3 IMP sequence), the self-measured Living Asset Registry state is:

| Asset                                               | Count | Size       |
| --------------------------------------------------- | ----- | ---------- |
| `mission-history.yaml` timeline entries             | 42    | 2480 lines |
| `.mission/snapshots/YYYY-MM-DD_vX.Y.Z_mission.yaml` | 42    | 316 KB     |
| `.mission/decisions/MDR-NNN-*.md`                   | 8     | ~100 KB    |
| `.mission/evals/*.result.yaml`                      | 1     | 8 KB       |
| `.mission/evidence/VERIFICATION_LOG.yaml`           | 1     | 12 KB      |
| `.mission/` total                                   | —     | 496 KB     |

Growth is approximately linear with release cadence. Extrapolated at the 2026-04-21 rate (7 releases / day during the IMP sequence sprint — not a steady state), 6 months of continuous cadence would push `mission-history.yaml` past 10k lines and `snapshots/` past 5 MB. In the long-run steady state (estimated ≤ 10 releases / month after IMP closure) projections soften, but the governance gap is the same: **no existing MDR states what Mission Spec owes, keeps, compacts, or discards**.

Three prior MDRs partition the governance surface:

- **MDR-005** — what code is allowed to live in `src/` (integrity verifiers / living-asset maintainers / distribution surface; orchestration and runtime out).
- **MDR-006** — how releases are versioned (consumer-surface-based MAJOR/MINOR/PATCH).
- **MDR-007** — how docs stay in sync across locales (EN + KO + ZH same-commit policy).

**Growth of the registry itself is the un-governed 4th axis**. qmonster-style adopters will eventually hit the same question: "when does the registry become too large to load, search, or clone?" Addressing this before an adopter encounters the wall — while Mission Spec's own registry is still under 500 KB — is cheaper than retrofitting a compaction workflow post-bloat.

Reviewer input (cross-session Rev.4 Codex §7 "Low priority" note from v1.16.0-series): "accumulated registry size is not yet a bottleneck, but no policy covers it — an adopter with a decade of history would have no guidance." Folding that concern into a governance MDR closes the gap.

## Decision

Adopt a **compact-at-phase-transition** policy governed by a single invariant and five per-asset retention rules.

### §INVARIANT — Reproducibility preservation

Any compacted asset MUST be reconstructable from the combination of:

1. **Git history** of the repo (commits + `related_commits` references are authoritative).
2. **The remaining Living Asset Registry** (kept entries + decisions + evolution_summary).

No compaction step may destroy information that cannot be recovered from (1) + (2). This is tested by the existing `reconstruction-verifier` + `validate-history-commits` invariants at commit time; compaction PRs must pass both before merge.

### §TIMELINE — `mission-history.yaml` timeline entry retention

- Detailed timeline entries for the **current phase** (per `evolution_summary.phases[last]`) are kept verbatim.
- When a new phase begins (phase identity transition — e.g., the v1.7.0 "architecture-assetization" phase shift), **prior phases' entries MAY be compacted** into the corresponding `evolution_summary.phases[]` block + `key_decisions[]` index.
- Each compacted entry's original `change_id`, `semantic_version`, `date`, and `related_commits` MUST survive as a single line inside the phase summary OR as a preserved stub entry (`persistence: "summarized"`).
- Compaction is **manual**: the maintainer opens a "registry compaction" chore release at phase boundary, and the resulting diff is auditable.

### §SNAPSHOTS — `.mission/snapshots/` retention

- **All phase-boundary snapshots** (version that ends a phase, e.g., `v1.6.1_mission.yaml` for "eval-extensibility" closure) are preserved permanently.
- **Intra-phase snapshots** (e.g., the 10+ snapshots between v1.16.0 and v1.16.19) MAY be pruned when the phase closes, keeping only phase-boundary versions + the current phase's full set.
- Snapshot file content is recoverable via `git show <commit>:mission.yaml` if needed, so pruning is information-preserving under the invariant.
- `mission-spec snapshot --retroactive` (IMP-8 candidate) would be the inverse operation — regenerating intra-phase snapshots from git history.

### §EVIDENCE — `.mission/evidence/VERIFICATION_LOG.yaml`

- Verification evidence for the **current phase** is appended release-by-release.
- Prior phases' evidence MAY be replaced with a phase-summary entry (`evidence_summary` stanza) citing the last recorded `npm test` count, coverage, and validator matrix at phase close. The detailed per-release entries are git-reachable via the phase-closing commit.

### §EVALS — `.mission/evals/*.result.yaml` retention

- LLM-eval override files (`<name>.result.yaml`) are keyed to specific `evals[].name` in the current `mission.yaml`.
- When a `done_when` entry using that eval is removed or renamed (schema v2 migration, policy shift), the orphan override file MAY be deleted.
- Since eval overrides are fixture-like (`passed: bool + reason`), git history retains full recoverability.

### §DECISIONS — `.mission/decisions/MDR-NNN-*.md` retention

- **MDRs are permanent**. MDR trigger rules (see MDR_TEMPLATE.md) already bound creation rate, so unbounded growth is unlikely.
- Superseded MDRs get `Status: Superseded by MDR-NNN` headers rather than deletion — the audit chain must remain traversable.

## Rationale

1. **Phase-boundary semantics beat arbitrary thresholds.** Time-windowed ("keep last 90 days") or size-windowed ("compact when file > 1 MB") retention are cheap to implement but decouple compaction from the project's own meaningful transitions. Mission Spec's `evolution_summary.phases` already names semantic boundaries; aligning compaction with those boundaries means the resulting compacted registry still tells the same story, just at one zoom level up.

2. **Reproducibility invariant is the real constraint, not file size.** Mission Spec's identity since v1.7.0 is "재현 가능한 아키텍처 자산 도구". Any retention policy that could drop reconstruction would contradict identity; framing the rule as an invariant-first policy prevents well-meaning cleanup PRs from crossing the line.

3. **Manual-trigger compaction is YAGNI-aligned.** Automating phase-boundary compaction prematurely (before Mission Spec's own registry even approaches a bloat threshold) would commit tool-maintenance burden for a non-urgent problem. A future `mission-spec compact` subcommand is an IMP-9 candidate but explicitly not scheduled by this MDR.

4. **Adopter playbook, not adopter enforcement.** MDR-008 tells adopters _what to do_ when their Living Asset Registry grows past comfort — it does NOT add validators that fail adopter builds. This matches MDR-005 (permissive scope rules) and MDR-007 (trilingual policy applies to Mission Spec itself, not to adopter documentation choices).

5. **Git as fallback truth.** Mission Spec is a Git-resident artifact; the policy leans on git history to keep compaction reversible. Adopters who abandon git for Mission Spec are already off the supported workflow; this MDR doesn't extend its reach beyond the git-backed happy path.

## Consequences

- **Mission Spec's own registry stays uncompacted at v1.19.0.** The 42-entry timeline, 42-file snapshots/, and 8-MDR decisions/ are all below any reasonable threshold. Compaction becomes relevant at the next phase transition (candidate: v2.0 if/when `done_when` evaluator gets an architectural rewrite).
- **`IMP-9` candidate opens**: `mission-spec compact <phase>` CLI that executes §TIMELINE + §SNAPSHOTS rules on demand, emits a diff, and refuses to apply if `reconstruction-verifier` fails on the proposed state. Not scheduled.
- **`mission-history.yaml` schema gains no breaking change.** `persistence: "summarized"` is a new allowed enum value for stub entries; this is additive and backward-compatible.
- **Adopter guidance added** via the relevant `README.md` "Living Asset Registry" section (future PATCH) pointing at MDR-008.
- **Schema v2 migration** (hypothetical) would likely combine MDR-008 compaction with breaking timeline-entry shape changes — handled by existing `core/migration.ts` infra.

## Alternatives Considered

- **Unbounded growth (do nothing).** Rejected: the gap was flagged in a prior adversarial review; explicit governance beats "trust text is cheap" especially when a decade-scale adopter eventually appears.
- **Time-windowed retention (e.g., "keep last 12 months").** Rejected: decouples compaction from project semantics. A project in a stable phase for 3 years shouldn't have its phase record pruned just because time passed.
- **Size-windowed retention (e.g., "compact when file > 2 MB").** Rejected: same issue — size thresholds are project-agnostic and would cause churny mid-phase compactions. Also encodes a number that will date badly.
- **Automated compaction on every commit.** Rejected: compaction is a semantic decision (which phase boundary?) that requires maintainer judgment. Automating it would either be wrong often or reduce to a manual trigger with a script wrapper — the latter is IMP-9 candidate.
- **Validator that fails builds when registry exceeds a threshold.** Rejected: punishes adopters for project duration. MDR-008 is guidance, not enforcement.
- **Schema-level `expires_at` field on timeline entries.** Rejected: encourages arbitrary date-based expiry; doesn't survive the invariant review. Git dates already provide temporal context.
